🆚 Why pytest over unittest?
Below is a breakdown of why pytest is preferred in most modern Python projects:

1️⃣ Much simpler syntax
unittest requires classes and boilerplate code.
pytest allows simple functions — no classes required.

--------------------------------------
Example: Same test in both frameworks
👉 Using unittest

import unittest

class TestMath(unittest.TestCase):

    def test_add(self):
        self.assertEqual(1 + 2, 3)

if __name__ == '__main__':
    unittest.main()

--------------------------------------

👉 Using pytest

def test_add():
    assert 1 + 2 == 3

🔹 No classes
🔹 No boilerplate
🔹 Just simple functions



2️⃣ Auto-discovery of tests
pytest automatically finds tests based on:

File name starting with test_
Function name starting with test_

No need to manually register tests like in unittest.

------------------------------------------------------------------------------------------------------------------------------------------------------------


3️⃣ More powerful assertions
In unittest, you use methods like:

assertEqual()
assertTrue()
assertIn()
…

In pytest → just use normal assert, which gives excellent error messages.

def test_numbers():
    assert (5 + 5) == 11    # pytest shows exact difference on failure
``

4️⃣ Fixture support
pytest fixtures are powerful for creating reusable test setup, such as:

DB connection setup
API client session


6️⃣ Better test organization
pytest supports:

Test classes
Test modules
Test directories
Shared fixtures

Very easy to scale for large automation projects.

🔥 Usage of Test Suites in pytest
Here are the main use-cases of suites in pytest.

1️⃣ Run the entire test suite

pytest

Pytest will collect all tests and run them as one suite.

2️⃣ Run a specific file (partial test suite)

pytest test_math.py

This runs only the tests inside test_math.py.

3️⃣ Run tests inside a directory (modular suite)

pytest tests/api/

Useful in microservices and DevOps CI/CD pipelines.

4️⃣ Run a specific test class

pytest test_math.py::TestOperationsShow 

5️⃣ Run a specific test function

pytest test_math.py::test_add

ABOUT FIXTURES
--------------

import pytest

# Define a fixture
@pytest.fixture
def sample_data():
    return {"a": 1, "b": 2, "c": 3}

# Use the fixture in a test
def test_key_exists(sample_data):
    assert "a" in sample_data

def test_value_check(sample_data):
    assert sample_data["b"] == 2




🔍 How It Works:
@pytest.fixture marks the function as a fixture.
sample_data is passed as an argument to the test functions.
Pytest automatically calls the fixture and injects its return value into the test.



Based on the Pytest in pytest.ini file we can add multiple markers and also we add the "addopts" which helps for showing the maxfailure in test file based on that the file stops after reaching certain failures of test, and for class we use python_classes = *Suite this will run all the classes that contains "Suite" in their class name.

20. What is a test coverage report, and how do you generate it in Pytest?
A test coverage report shows how much of your code is being covered by your tests.It is produced by a coverage tool, such as Coverage.py, which is integrated into Pytest. The --cov option and the path to the directory containing your code must be used in Pytest in order to generate a coverage report. For example:

pytest --cov=my_module tests/

12. What is monkey patching in Pytest, and how do you use it?
In Pytest, monkey patching is the practice of swapping out a function or variable for one that is particular to a given test. When you need to change a function's behavior to make it simpler to test, this can be helpful. Using the monkeypatch fixture, which offers ways to change properties and functions, monkeypatching is accomplished in Pytest. For example:

def test_my_function(monkeypatch):
    def mock_function():
        return "mocked result"
    monkeypatch.setattr("my_module.my_function", mock_function)
    assert my_module.my_function() == "mocked result"

The short version


Parametrized fixture (@pytest.fixture(params=...))
Parametrizes setup data/behavior. Each test that depends on this fixture will run once per parameter. Think: “Run all tests against each browser.”


Parametrize marker (@pytest.mark.parametrize(...))
Parametrizes function inputs (arguments of the test function). Think: “Run this specific test with multiple input tuples.”

pytest.ini is a configuration file that defines default settings for pytest test runs, removing the need to repeatedly pass long command-line arguments. It is placed in the root directory of your project and takes precedence over other supported config files like pyproject.toml, tox.ini, or setup.cfg.

By centralizing options, it ensures consistent test behavior across environments, CI/CD pipelines, and developer machines.

Example basic structure:

[pytest]
minversion = 7.2
addopts = -ra -q
testpaths = tests
Copy
This sets the minimum pytest version, default CLI options, and test discovery paths.

Common uses of pytest.ini:

Logging configuration Control log output format, level, and date format globally:

[pytest]
log_cli = true
log_level = DEBUG
log_format = %(asctime)s %(levelname)s %(message)s
log_date_format = %Y-%m-%d %H:%M:%S
Copy

[pytest]
